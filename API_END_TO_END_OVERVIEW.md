# Gym Management API End-to-End Overview

This document gives a human-friendly overview of the Gym Management API from end to end. It is written for developers, mobile app teams, QA testers, and non-technical stakeholders who need to understand what each module does and how the APIs relate to each other.

The project is a Next.js App Router application. API handlers live under `app/api`, shared logic lives under `lib`, and the database model is defined in `prisma/schema.prisma`.

## Big Picture

The system has three main API surfaces:

| Area | Base path | Main users | Purpose |
| --- | --- | --- | --- |
| Admin API | `/api/v1` | Super Admin, Manager, Auditor | Manage gym data, payments, settings, reports, and access control. |
| Member Mobile API | `/api/mobile` and `/api/mobile/member` | Gym members | Login, view subscriptions, renew/upgrade, pay with Waafi, see workouts, schedules, and notifications. |
| Trainer Mobile API | `/api/mobile/trainer` | Trainers | Login, manage profile, view assigned members/groups, create workouts and schedules. |

## Authentication Summary

### Admin Authentication

Admin users sign in through NextAuth credentials:

- `GET/POST /api/auth/[...nextauth]`
- `POST /api/auth/forgot-password`
- `POST /api/auth/reset-password`

Admin routes use `requireAuth()` or `requirePermission(permissionName)`.

| Method | Endpoint | Description |
| --- | --- | --- |
| `GET` | `/api/auth/[...nextauth]` | NextAuth route handler for session/auth callbacks. |
| `POST` | `/api/auth/[...nextauth]` | NextAuth credentials sign-in handler. |
| `POST` | `/api/auth/forgot-password` | Sends admin password reset code if the email exists. |
| `POST` | `/api/auth/reset-password` | Resets admin password using email, code, and new password. |

Important behavior:

- Sessions are JWT-based.
- Permissions are loaded from Access Control roles.
- `Super Admin` bypasses normal permission checks.
- Admin password reset sends a short-lived reset code by email.

### Mobile Authentication

Mobile users use a separate token system:

- `POST /api/mobile/auth/login`
- `GET /api/mobile/auth/me`
- `POST /api/mobile/auth/logout`
- `POST /api/mobile/auth/forgot-password`
- `POST /api/mobile/auth/reset-password`

Mobile token format is a signed token generated by `lib/mobile-auth.ts`. It contains:

- account id
- role: `MEMBER` or `TRAINER`
- expiration time

Important behavior:

- Mobile tokens are sent as `Authorization: Bearer <token>`.
- Member APIs reject trainer tokens.
- Trainer APIs reject member tokens.
- Inactive or suspended mobile accounts cannot use protected mobile routes.
- Phone login supports common Somalia formats such as `061...`, `61...`, and `25261...`.

## Shared Response Style

Most APIs return JSON. Mobile APIs usually include:

```json
{
  "success": true
}
```

Errors are handled through the shared error handler and usually look like:

```json
{
  "error": "Message"
}
```

Pagination uses:

```json
{
  "pagination": {
    "total": 100,
    "page": 1,
    "limit": 10,
    "pages": 10
  }
}
```

## Important End-to-End Flows

### 1. Admin Creates a Member

1. Admin calls `POST /api/v1/members`.
2. The system creates the member.
3. If an initial plan is included, the system creates a subscription and payment.
4. The system creates a mobile account for the member.
5. A welcome email is attempted if email is available and SMTP is configured.
6. Activity logs are written.

### 2. Admin Creates a Trainer

1. Admin calls `POST /api/v1/trainers`.
2. The system creates the trainer.
3. The system creates trainer mobile login credentials.
4. A welcome email is attempted.
5. Activity logs are written.

### 3. Member Renews or Upgrades

1. Member logs in with `POST /api/mobile/auth/login`.
2. Member views plans using `GET /api/mobile/member/plans`.
3. Member creates a pending upgrade or renewal:
   - `POST /api/mobile/member/subscription/upgrade`
   - `POST /api/mobile/member/subscription/renew`
4. Member pays through Waafi:
   - `POST /api/mobile/member/payments/waafi/initiate`
5. Successful payment activates the subscription and member status.
6. A confirmation notification is created.

### 4. Trainer Assigns Workout and Schedule

1. Trainer logs in with `POST /api/mobile/trainer/auth/login`.
2. Trainer views assigned members or groups.
3. Trainer creates a workout for exactly one member or one group.
4. Trainer creates a schedule linked to a workout.
5. Target members receive notifications.
6. Members see assigned content through member workout and schedule APIs.

### 5. Push Notification Flow

1. Member or trainer logs in.
2. Mobile app registers FCM token using `POST /api/mobile/device-tokens`.
3. Member tokens subscribe to member topics.
4. Trainer tokens subscribe to trainer topics.
5. Admin-created notifications attempt Firebase push after saving the database record.

## WaafiPay Behavior

WaafiPay is used by both admin and member mobile payment flows.

Key points:

- Configuration is stored in `AccessControlSettings`.
- API key is never returned in frontend-safe responses.
- Somalia phone numbers are normalized to `252...`.
- Supported providers are `EVC_PLUS`, `JEEB`, `ZAAD`, `SAHAL`.
- Payment attempts are saved before the remote request.
- Failed validation/config/request attempts are recorded with failure details.
- Raw Waafi response is saved into `Payment.rawResponse`.
- Successful online payments activate the linked subscription and member.
- Successful payment creates an `UPGRADE_CONFIRMATION` notification.
- Failed payment creates a `PAYMENT_REMINDER` notification.

## Firebase Push Behavior

Firebase is optional and configured from settings.

Key points:

- Mobile apps register device tokens at `/api/mobile/device-tokens`.
- Member topic: `members`
- Trainer topic: `trainers`
- Account-specific topics are also returned.
- Admin notification creation saves the notification first, then attempts push.
- Push failure does not stop notification record creation.
- Firebase test route performs a dry-run validation.

## Security Notes

- Admin routes are permission-protected.
- Mobile routes are role-protected.
- Trainer mobile routes enforce ownership before returning members, groups, workouts, or schedules.
- Member mobile routes only return data owned by the logged-in member.
- Password hashes are never returned.
- Mobile temporary passwords are encrypted for privileged admin fallback viewing.
- Waafi API key is hidden from frontend-safe responses.
- Email reset flows use short-lived hashed reset codes.
